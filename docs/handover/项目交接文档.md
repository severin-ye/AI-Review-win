## 1. 项目是什么

**AI 文档审校桌面软件**：上传 Word 文稿 → AI 自动审校（事实/术语/语法/格式四类问题，附医学知识库证据）→ 人工逐条确认 → 导出双版本 Word 成稿。目标用户是医学文稿编辑场景。

技术形态：**Electron 桌面壳（主进程 + preload）+ React 渲染层 + Python FastAPI sidecar 后端（本地推理/RAG/LLM 调用）**，另含一套完整的**局域网许可证授权系统**（授权端 license-server + 客户端激活/心跳/离线验签）。

历史：3 年老项目（原 Python/Tkinter，保留在 `legacy/code` 只读参照），**2026-07 按「方案 D」目录结构全面重构**（设计文档：`docs/design/AI-Review-Electron-重构设计.md`）。当前活跃代码全部为重构后的 Electron 版。

## 4. 怎么跑起来（开发）

```bash
# 0. 一次性：装依赖
npm.cmd install
npm.cmd run backend:setup        # 建 app/server/.venv 并装 app/server/requirements.txt
app/server/.venv/Scripts/python.exe -m pip install -r app/license-server/requirements.txt

# 1. 起许可证服务器（授权端，另开终端；开发期源码运行）
cd app/license-server
../server/.venv/Scripts/python.exe run.py
# → admin http://127.0.0.1:8767/（自动开浏览器）、employee 0.0.0.0:8768
# → 数据写入 app/license-server/.data（首次启动生成 DEV 密钥对）

# 2. 起桌面端（仓库根）
npm.cmd run dev                  # electron-vite dev：主进程+preload+renderer，自动拉起后端 sidecar
```

**激活流程**：客户端启动进激活页 → 填服务器地址（本机开发 `http://127.0.0.1:8768`，局域网填 `http://<老板机IP>:8768`）+ 许可证密钥（管理页创建）→ 激活成功进主界面。凭证加密存 `%APPDATA%\ai-review\license.dat`，之后启动免激活。

**公钥配对规则（简版）**：服务器的签名密钥对在其 `.data/keys/` 里（私钥 DPAPI 保护）；客户端验签公钥是**构建期焊死**的 `app/desktop/resources/license-public.pem`（由 `app/license-server/scripts/export_public_key.py` 从服务器导出）。**换密钥 = 重新导出公钥 + 重打员工包**，否则老客户端验签失败。详细操作见 `docs/license/runbook.md`。

## 5. 怎么构建分发

**客户端（Electron 桌面应用，两步）**：

```bash
npm.cmd run dist:backend   # PyInstaller：app/server/dist/ai-review-backend.exe（用 app/server/.venv）
npm.cmd run dist           # electron-vite build + electron-builder → release/ 下 NSIS 安装包
```

- electron-builder 配置在 `electron-builder.yml`：图标 `build/icon.ico`；`extraResources` 带 `ai-review-backend.exe` 和 `license-public.pem`；NSIS 非一键安装、可改目录、建桌面/开始菜单快捷方式。
- 模型文件（数 GB）**不打包**：首次启动后由设置页从 hf-mirror 预下载（断点续传）。

**授权端（许可证服务器 exe）**：

```bash
cd app/license-server
../server/.venv/Scripts/python.exe -m PyInstaller ai-review-license-server.spec --noconfirm
# → dist/句读授权中心.exe（图标 assets/icon.ico）
```

- frozen exe 的数据目录 = **exe 旁的 `.data/`**（SQLite + 密钥，备份整个 .data 即可）。
- exe 首次启动会弹一次 UAC（自动加防火墙放行规则；用户取消只告警不阻断）。
- spec 里 `hiddenimports` 收集 `pkg_resources`/`jaraco.*`/`platformdirs` 等外部化依赖是**踩过坑后保留的**，删减会导致 frozen 启动失败。

## 6. 许可证系统速览

- **角色**：老板机跑 license-server（admin 8767 管理页发密钥/吊销/看设备；employee 8768 对员工机提供激活/心跳/刷新 API）；员工机跑客户端，只含公钥。
- **信任模型**：激活/刷新下发 **Ed25519 签名的离线凭证 token**（含 license_id、device_id、expires_at、features、license_version）；心跳响应也带签名。客户端用内嵌公钥验签，**不手写密码学**（canonical JSON 序列化 + cryptography 库）。
- **流程一段话**：员工端输入服务器地址+密钥 → activate 绑定设备（一证一设备，按全局 device_id 查重）→ 拿签名 token 落盘（safeStorage 加密）→ 之后每次启动本地验签放行 + 后台心跳续期；断网在宽限期内离线可用；服务器吊销 → 下次心跳/冷启动联网即锁；到期前按阈值（3 天/1 天/1 小时）提醒。
- **命根子**：服务器的 `.data/`（密钥 + SQLite）。丢密钥 = 全部客户端作废；**公钥焊在客户端**，换密钥必须导出公钥重打员工包（`docs/license/runbook.md` 有仪式流程）。
- **深入阅读**：`docs/license/` 六件套（见 §10）。

## 7. 测试体系

三套单测/集成测试 + 两个脚本化端到端设施，全部命令已实跑核对：

| 套件 | 规模 | 运行命令 |
| --- | --- | --- |
| 许可证客户端单测（vitest） | **63 passed** | 仓库根：`npm.cmd run test:license` |
| 老板端 license-server（pytest） | **72 passed** | `cd app/license-server && ../server/.venv/Scripts/python.exe -m pytest tests/ -q` |
| 审校后端 app/server（pytest） | **30 passed** | `cd app/server && .venv/Scripts/python.exe -m pytest -q` |

- vitest 只跑 `app/desktop/main/license/__tests__/`（见 `vitest.config.ts`）：Node 环境、全依赖注入 + 假 fetch，不碰真服务器；含「激活推送与失败状态」回归（修复激活失败提示被吞的 bug）。
- license-server pytest：每个用例独立临时 `.data`（`AI_REVIEW_LICENSE_DATA_DIR`）+ 可注入时钟；`test_singleinstance.py` 的单实例用例 **mock ctypes，不创建真实全局 Mutex**。
- **许可证端到端冒烟** `scripts/license-smoke.ts`（不进 vitest 默认套件）：对拍向量/激活/心跳/撤销/断网等全链路，需要真实 license-server 跑在 8767/8768。打包与运行方式见脚本头部注释（esbuild 打成 `scripts/.license-smoke.bundle.mjs` 后 `node` 执行；产物已入 .gitignore）。
- **Electron shell e2e** `scripts/e2e-shell-entry.ts`：真实主进程 `index.ts` 全量逻辑（仅 sidecar 换本地桩）+ 真实 preload/renderer 构建产物 + 临时 userData，覆盖 splash、托盘、单实例（会真实拉起第二实例进程验证抢锁失败退出 + 第一实例窗口被激活）。用法：`node scripts/build-e2e-shell.mjs`（产物 `out/main/e2e-shell.cjs`，会被 `npm run build` 清掉需重建）→ `node_modules/.bin/electron out/main/e2e-shell.cjs`（输出 `E2E_SHELL_RESULT` JSON，退出码 0 为通过）。
- **license-server 双开实测** `app/license-server/scripts/dual_instance_check.py`：用打包后的 exe + 89xx 端口 + 临时 `.data` 实测「第二实例被拒 + 第一实例存活」，自动清场。用法见 §9 与脚本 docstring。

质量门禁（提交前必过）：`npm.cmd run typecheck` + `npm.cmd run build` + `npm.cmd run test:license` + license-server pytest。

## 11. 常见坑

1. **双实例事故（2026-07-18）**：两个 license-server exe 并发启动互相覆盖 `.data` 密钥。现在两端都有单实例锁，但**测试双开必须隔离**：用 `AI_REVIEW_LICENSE_ADMIN_PORT`/`EMPLOYEE_PORT` 换 89xx 端口 + `AI_REVIEW_LICENSE_DATA_DIR` 指临时目录 + `AI_REVIEW_LICENSE_SKIP_FIREWALL=1`（直接用 `app/license-server/scripts/dual_instance_check.py`）。**绝不**在 8767/8768 真实实例运行时起第二个默认端口实例、绝不杀它的进程、绝不动它的 `.data`。
2. **PyInstaller `__file__` 临时目录坑**：frozen exe 里 `__file__` 指向 `_MEIxxxx` 临时解压目录，**进程退出即被清除**——密钥/数据库绝不能按 `__file__` 定位。已修：`config.py` 的 `_default_data_dir()` 在 frozen 下用 `sys.executable` 旁 `.data`。新增文件定位逻辑时沿用这个模式。
3. **`.gitignore` 的 `build/` 收窄原因**：根 `build/` 存着 electron-builder 图标（必须入库），所以只忽略 `app/server/build/` 和 `app/license-server/build/`（PyInstaller workpath）。注意 `dist/` 与 `*.spec` 被全局忽略，但两个 spec（`ai-review-backend.spec`、`ai-review-license-server.spec`）已显式豁免入库。
4. **safeStorage 降级路径**：客户端凭证加密（`license/storage.ts`）与设备 ID（`license/deviceId.ts`）依赖 Electron safeStorage；safeStorage 不可用时走降级路径并记告警日志（不要假设一定可用）。细节见 `docs/license/security.md`。
5. **PyInstaller onefile exe 杀进程要杀进程树**：`Popen.terminate()`/任务管理器只杀 bootloader（外壳），真正干活的子进程会变孤儿继续占端口；必须 `taskkill /PID <pid> /T /F`。Git Bash 里正斜杠参数会被路径转换，要写 `taskkill //PID <pid> //T //F`。
6. **frozen 启动失败的 pkg_resources 依赖**：setuptools 78 后 `jaraco.text`/`platformdirs` 等不再 vendored，license-server spec 里的 `collect_submodules` 清单是修好的，别精简。
7. **e2e 产物生命周期**：`out/` 会被 `npm run build` 清空——`out/main/e2e-shell.cjs` 需在 build 后用 `node scripts/build-e2e-shell.mjs` 重建；`scripts/.e2e-*.cjs`、`scripts/.license-smoke.bundle.mjs` 都是临时产物（已入 .gitignore 或可自行删除）。
