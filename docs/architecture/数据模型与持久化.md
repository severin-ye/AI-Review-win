# 数据模型与持久化

> 代码位置：`app/server/app/models/tables.py`（SQLModel/SQLAlchemy 2.0 声明式模型）、`app/server/app/core/db.py`（引擎与迁移）。
> 本文逐表列出全部字段、默认值与索引，并说明磁盘目录布局、模型文件与导出文件的持久化约定。

## 1. 数据目录

解析优先级（`app/server/app/core/config.py`）：

```text
1. 环境变量 AI_REVIEW_DATA_DIR          （测试用它隔离到临时目录）
2. AI_REVIEW_PACKAGED=1 → %APPDATA%/ai-review   （Electron sidecar 生产模式注入）
3. 默认 app/server/.data/               （开发模式）
```

实测开发模式目录布局：

```text
app/server/.data/
├── app.db                          # SQLite 主库（11 张表）
├── kb/
│   ├── files/<kb_id>.<ext>         # KB 原始上传文件
│   ├── lancedb/                    # LanceDB 向量库（表名 kb_chunks）
│   └── bm25.pkl                    # BM25 索引（rank_bm25 序列化）
├── models/                         # 本地模型（自动探测，见 §4）
│   ├── bge-m3/          2.2 GB     # pytorch_model.bin = 2,271,145,830 B
│   ├── sat-3l-sm/       816 MB
│   └── xlm-roberta-base/ 14 MB
└── projects/<doc_id>/              # 每个文档一个目录
    ├── original.docx               # 原始上传件
    ├── no_table.docx               # 剥离审校表格后的工作副本
    ├── parsed.md                   # docx 解析产物（pandoc 风格 markdown）
    ├── tables.docx                 # 提取出的审校表格（可选）
    └── exports/                    # 导出产物（见 §5）
```

### 2.8 kb_chunks — 已定义但业务未使用 ⚠️

SQL 侧定义了 `kb_chunks` 表（id / kb_document_id FK / idx / text / created_at），**但当前实现不写入该表**——chunk 的唯一持久化来源是 LanceDB 表（见 §3）。此表为早期设计残留，保留不影响运行；新代码不要依赖它。

## 3. LanceDB 向量库

- 位置：`<data_dir>/kb/lancedb/`，表名固定 `kb_chunks`。
- Schema：`chunk_id`（string，`sha256(hash:idx)[:16]`）、`kb_document_id`（string）、`idx`（int32，文档内块序号）、`text`（string）、`source_name`（string）、`vector`（`list<float32>(dim)`，dim 随 embedding 模型，bge-m3 为 1024）。
- **维度不符自动重建**：打开表时若发现 vector 维度与当前 embedding 模型不一致，删表重建（防止换模型后检索崩溃）。
- BM25 索引单独序列化在 `<data_dir>/kb/bm25.pkl`，与 LanceDB 同步重建。

## 4. 本地模型探测

`app/server/app/core/config.py` 的探测顺序（对 sat-3l-sm / bge-m3 / xlm-roberta-base 三个模型各自独立）：

```text
1. 对应环境变量（AI_REVIEW_SAT_MODEL_PATH / AI_REVIEW_SAT_TOKENIZER_PATH /
   AI_REVIEW_EMBEDDING_MODEL_PATH）——存在即采用
2. <data_dir>/models/<dirname>/ —— 目录存在即离线加载
3. 都没有 → None，由 HuggingFace Hub 在线下载到 ~/.cache/huggingface
   （可用 HF_ENDPOINT 环境变量指向镜像）
```

实测大小（`<data_dir>/models/`）：bge-m3 2.2 GB、sat-3l-sm 816 MB、xlm-roberta-base 14 MB。下载任务由 `model_download` 类型的 job 承载，SSE 汇报文件级进度。

## 5. 导出产物

- 目录：`<data_dir>/projects/<doc_id>/exports/`（或用户自定义 `output.dir`）。
- 命名：`{原文件名stem}_审校修订1_.docx`（成稿）/ `{stem}_审校修订2_.docx`（标注稿）等。
- `documents.exports_json` 记录最近一次导出：

```json
{
  "clean":   "...exports/xxx_审校修订1_.docx",
  "marked":  "...exports/xxx_审校修订2_.docx",
  "adopted": "...exports/xxx_审校修订对照.docx",
  "warnings": [],
  "exported_at": "2026-01-01T12:00:00"
}
```

三个键分别对应：成稿（应用全部 accepted/edited 修订）、标注稿（Word 批注形式）、审校对照（修订前后对照表）。

## 6. 轻量迁移 `_migrate`

`core/db.py` 在引擎初始化时执行幂等的 `_migrate`：用 `PRAGMA table_info` 检查并 `ALTER TABLE ADD COLUMN` 后补列（目前是 `documents.exports_json` 与 `jobs.kb_document_id`）。无 Alembic——表结构演进靠「新增列 + 默认值」完成，删除/改型需手工处理。

## 7. 级联与孤儿清理

- documents → blocks / sentences：CASCADE 删除。
- sentences → queries / evidence / corrections：CASCADE 删除。
- jobs → job_events：CASCADE 删除。
- 删除文档时同步清理 `<data_dir>/projects/<doc_id>/` 目录；删除 KB 文档时同步清理 `kb/files/<kb_id>.*` 并触发 LanceDB/BM25 重建。
- 后端启动时执行僵尸任务清理：把残留在 running 状态的 jobs 标记为 error（进程重启后不再有执行者）。
