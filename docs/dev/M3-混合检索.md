## 数据流

```
知识库文档(pdf/txt/csv/docx) → data/kb/files/<kb_id>.<ext>
  → 加载纯文本 → RecursiveCharacterTextSplitter(500/50)
  → chunk_id = sha256(content_hash:idx)[:16]（重切稳定，幂等 upsert）
  → EmbeddingProvider.embed 批量(L2 归一化) → LanceDB kb_chunks
  → 全量重建 BM25(jieba) → pickle data/kb/bm25.pkl
  → 全程事件写 job_events（SSE 推送前端进度）

文档句子（跳过参考文献块/占位符句）
  → ① 查询重写：chat_json 一次调用 → N=retrieve.query_count(默认8, 5-10截断) 问
       prompt：第1条保留原句 + 医学实体提取 + 多角度核查问题；失败/未配置 → [原句]
  → ② 全部问题一次批量 embed；每问并行两路：
       向量路 LanceDB cosine top-(3k)   BM25 路 jieba top-(3k)
  → ③ 每路跨问题 RRF：score=Σ1/(60+rank) → 各路 top-3
  → ④ 两路按 chunk_id 去重合并（≤6）→ evidence 表(source/rank/score/doc_name/chunk_text)
  → queries 表（重写问题溯源）→ documents.status = retrieved
```

## 配置项（settings 表，GET 含默认值）

| 键 | 默认 | 说明 |
|---|---|---|
| llm.base_url / llm.api_key / llm.model | 空 | OpenAI 兼容；未配置 api_key 时检索降级原句、chat_json 抛 LLMNotConfiguredError |
| retrieve.query_count | 8 | 查询重写数量，读取时截断到 5-10 |
| retrieve.vector_topk / retrieve.bm25_topk | 3 / 3 | 每句两路各取条数（融合前每问每路取 k*3 候选） |
| retrieve.rrf_k | 60 | RRF 常数 |
| retrieve.enabled | true | false 时 retrieve 接口返回 400（纯 LLM 审校路径，M4） |
| embedding.provider | local | local=BGE-M3 / openai=远端 / stub=测试隐藏档 |
| embedding.model | BAAI/bge-m3 | local=HF 名（或 AI_REVIEW_EMBEDDING_MODEL_PATH 本地目录覆盖）；openai=远端模型名 |
| segment.review_references | false | true 时参考文献块也参与检索/审校 |

## 设计取舍记录

1. **SQLite kb_chunks 表未使用**：设计文档 §6 原方案是 SQL 存文本 + LanceDB 外挂向量；实现简化为 LanceDB 单一来源（text 一并入列），重建 BM25 与检索都无需回查 SQL。SQL 表保留不删（M1 已建）。
2. **BM25Retriever 不整体 pickle**：其内含 `_thread.RLock` 无法序列化；改为 pickle `{vectorizer: BM25Okapi, docs}`、加载时重组（仍是 LangChain 组件，jieba preprocess_func 不变）。
3. **BM25 微小语料退化**：rank_bm25 的 Robertson IDF `ln((N-n+0.5)/(n+0.5))` 在词项出现于恰好一半文档时为 0（2 块语料全零分）。真实知识库（数十块以上）无此问题；单测用 ≥4 块语料规避。
4. **去重冲突仲裁**：同一 chunk 被两路命中时，取 RRF 分高的一路定 source 标签（平分局倾向 vector 先写入），最终按融合分降序，总量 ≤6。
5. **LanceDB 维度变更**：embedding 维度与已建表不符时自动删表重建（如 stub↔BGE-M3 切换），需重新索引知识库（reindex 一键完成）。
6. **jobs 表新增 kb_document_id 列**（轻量 ALTER 迁移，同 M2 is_reference 模式），知识库索引任务独立于文档流水线任务。

## 已知问题 / 后续注意

1. **Electron sidecar 不带 embedding 环境变量**：与 M2 已知问题 1 相同——Electron 拉起的后端默认读不到 `AI_REVIEW_EMBEDDING_MODEL_PATH`，local provider 会尝试 HF 下载。M5 模型管理（GET /api/models/status、预下载）应把本地目录方案正式化；过渡期可在设置里切 `embedding.provider=openai`。
2. **检索为同步执行**：POST retrieve 会阻塞至全文档句子完成（每句 1 次 LLM 重写 + 两路检索）。长文档（数百句）耗时主要花在 LLM 重写调用上；进度经 job_events/SSE 可见，M4 如需可改后台线程 + 前端轮询。
3. **PDF 加载器未经单测覆盖**（单测不构造二进制 PDF；pypdf 对扫描件提取为空会报「未切出有效文本块」错误并置 failed，属预期行为）。
4. **BM25 微小语料 IDF 退化**：见「设计取舍记录 3」。知识库只有 1-2 个短文档时关键词路区分度差，向量路兜底。
5. **llm.api_key 明文存 settings 表**：设计文档 §8 的 keytar 钥匙串方案未实施（桌面单机场景，M5 打磨时可加）。
6. 根目录 `tsconfig.json` 的 `baseUrl` 与新 TS 版本不兼容（TS5102），M1 起即存在；`npm run build`（electron-vite/esbuild）不受影响，故未改动根目录文件。
7. `langchain-community` 0.4.x 已宣布进入 sunset（官方引导迁往独立集成包）；当前 BM25Retriever 功能完整，后续版本留意迁移公告。
