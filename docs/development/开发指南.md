# 开发指南

> 面向开发者的实战手册：环境搭建、日常命令、测试、调试、打包与已知坑。
> 前提：已读 [README](../../README.md) 与《架构总览》。所有路径相对于仓库根目录 `AI-Review-win/`。

## 1. 环境要求（实测）

| 工具 | 实测版本 | 说明 |
|---|---|---|
| Node.js | v24.15.0 | 要求 ≥ 20.19（electron-vite 5 / Vite 7）；`.npmrc` 已配 Electron 国内镜像 |
| npm | 随 Node | 本仓库用 npm（package-lock.json），不要用 pnpm/yarn 混用 |
| Python | 3.12.13 | **必须用 `app/server/.venv` 里的解释器**，不要用系统 Python |
| Git Bash | — | Windows 下 npm 需用 `npm.cmd`（见 §7 坑 1） |

## 2. 首次搭建

```bash
# 1) 前端 + Electron 依赖
npm install

# 2) 后端虚拟环境与依赖（已封装为脚本）
npm.cmd run backend:setup
#    等价于：python -m venv app/server/.venv
#           app/server/.venv/Scripts/python -m pip install -r app/server/requirements.txt

# 3) PyTorch CPU 版需单独装（requirements 不含，避免拉 CUDA 大包）
app/server/.venv/Scripts/python -m pip install torch \
    --index-url https://download.pytorch.org/whl/cpu

# 国内加速可加清华镜像：
#   -i https://pypi.tuna.tsinghua.edu.cn/simple
```

## 3. npm scripts 逐条解释（package.json）

| 命令 | 实际行为 |
|---|---|
| `dev` | `electron-vite dev`：主进程/预加载/渲染层一并构建并启动 Electron；主进程拉起 Python sidecar（`app/server/.venv` 的 uvicorn，动态空闲端口，健康检查通过后开窗） |
| `dev:renderer` | `vite --config app/web/vite.config.ts`：只起渲染层 dev server，纯浏览器调试 UI（baseUrl 回退 127.0.0.1:8765，需另跑 `backend:dev`） |
| `build` | `electron-vite build` → `out/{main,preload,renderer}/`（生产构建，不经 tsc 类型检查） |
| `start` | `electron-vite preview`：以构建产物（out/）启动 Electron 预览 |
| `backend:dev` | `cd app/server && .venv\Scripts\python -m uvicorn app.main:app --reload --port 8765`，**不走 sidecar**——调后端最常用 |
| `backend:setup` | 建 venv + 装 requirements（见 §2） |
| `dist:backend` | PyInstaller（spec：`app/server/ai-review-backend.spec`）→ `app/server/dist/ai-review-backend.exe`（实测 379,184,269 B ≈ 362 MB） |
| `dist` | `npm run build` + electron-builder → `release/AI 审校助手 Setup <version>.exe`；**必须先手动跑过 `dist:backend`**（extraResources 要引用该 exe） |

典型日常组合：终端 A 跑 `npm.cmd run backend:dev`，终端 B 跑 `npm.cmd run dev:renderer`，浏览器开 vite 输出的地址 —— 无需 Electron 即可全栈联调。

## 4. 测试

```bash
cd app/server && .venv/Scripts/python -m pytest tests/ -q
```

30 个用例分布：

| 文件 | 数量 | 覆盖点 |
|---|---|---|
| `test_health.py` | 1 | /api/health 200 与版本字段 |
| `test_pipeline.py` | 4 | docx 解析→blocks/sentences 落库、parsed.md 产出、分段合并、幂等 |
| `test_rag.py` | 7 | chunk 切分、向量/BM25/RRF 融合、kb_chunks 维度重建、retrieve 接口、enabled=false 返回 400 |
| `test_review.py` | 11 | 审校 job 全流程、correction 落库、decide/batch、SSE 事件序列、断点续审 |
| `test_models.py` | 4 | 模型状态查询、路径探测优先级、下载 job、取消 |
| `test_export.py` | 3 | 成稿/标注稿/对照导出与 exports_json 记录 |

**隔离机制**（`tests/conftest.py`）：

- 导入任何 `app.*` 模块**之前**设置 `AI_REVIEW_DATA_DIR=tempfile.mkdtemp()`——SQLite/LanceDB/文件全部进临时目录；
- `AI_REVIEW_SEGMENTER=rule`——跳过 SAT 模型加载；
- embedding 走 `stub` provider——不下载 bge-m3；
- `monkeypatch` mock `chat_json`——不依赖真实 LLM；
- 数据库引擎在首次导入时绑定到该临时数据目录，整套测试进程内一致。

因此测试**完全离线、秒级完成**；新增用例沿用同一 conftest 即可。

## 5. 调试技巧

| 症状 | 排查手段 |
|---|---|
| 任务卡住/报错 | 查 `app/server/.data/app.db` 的 `jobs` 与 `job_events` 表（事件按 id 升序即时间线） |
| 想实时看 job 进度 | `curl -N http://127.0.0.1:8765/api/jobs/<job_id>/events`（SSE 流） |
| 查任务终态 | `curl http://127.0.0.1:8765/api/jobs/<job_id>` |
| 文档状态不对 | 查 `documents.status` 与 `error_message`；对照《后端详解》§3 状态机 |
| 文件产物缺失 | 检查 `app/server/.data/projects/<doc_id>/` 与 `kb/` 目录 |
| 模型没加载 | `curl http://127.0.0.1:8765/api/models/status`，对照《数据模型与持久化》§4 探测顺序 |
| LLM 不通 | 设置页保存后自动连通测试；或手工 curl `/api/settings/test-llm` |
| 前端连不上后端 | 浏览器预览模式看 BackendStatus 横幅；Electron 模式查开发者工具里 `window.api.getBackendUrl()` 的返回值 |

## 6. 打包发布

```bash
npm.cmd run dist:backend   # 先出后端 exe（PyInstaller，379,184,269 B ≈ 362 MB）
npm.cmd run dist           # electron-vite build + electron-builder → release/AI 审校助手 Setup 0.1.0.exe（≈ 459 MB）
```

- **安装包形态**（`electron-builder.yml` 的 nsis 段）：向导式安装（`oneClick: false`，下一步→自定义目录→完成）、按用户安装免管理员（`perMachine: false`）、自动创建桌面与开始菜单快捷方式。模型文件（数 GB）**不打包**，首次启动后经 设置页 → 模型管理 从 hf-mirror 预下载。
- **生产链路**：安装包 → Electron 主进程 → sidecar 注入 `AI_REVIEW_PACKAGED=1` + 动态端口 spawn `resources/backend/ai-review-backend.exe` → 数据目录切到 `%APPDATA%/ai-review`。
- **已端到端验证**（2026-07-17）：exe 单独跑通 /api/health；静默安装（`Setup.exe /S /D=...`）→ 双击启动 → 主进程与 sidecar 进程均在 → 静默卸载、快捷方式与目录清理干净。
- **jaraco 问题的解法**（曾是 M5 遗留）：setuptools 78 把 `pkg_resources` 的依赖外部化了，冻结 exe 里 vendored 机制失效。修法＝venv 里 `pip install jaraco.text platformdirs`（已写入 requirements.txt 打包段），并在 spec 的 hiddenimports 收集 `pkg_resources / jaraco.* / platformdirs / packaging / more_itertools`。
- **打包耗时参考**：PyInstaller 约 5–6 分钟（热缓存）；electron-builder 约 4–5 分钟（含 Electron/NSIS 下载，首次更久）。

## 7. 已知坑与注意事项

1. **Git Bash 里 `npm` 是 shell 包装脚本会报错，必须用 `npm.cmd`**（pnpm 无此问题）。
2. **后端必须用 `app/server/.venv` 的解释器**（3.12.13 实测）；系统 Python 版本不对或缺依赖会踩雷。
3. **torch 不在 requirements.txt 里**，必须单独按 CPU 源安装（§2），否则会尝试拉数 GB 的 CUDA 版。
4. `.npmrc` 的 `ELECTRON_MIRROR` 在 install 时可能打印镜像警告，**无害**。
5. **jaraco/pkg_resources 问题已修复**（见 §6）：旧 exe 报 `No module named 'jaraco'` / `platformdirs` 的解法已固化进 requirements.txt 与 spec。
6. **导出 docx 标题层级被扁平化为 Heading 1**（python-docx 重建阶段的已知损失）。
7. **块级段落合并后文本无损、但分段边界不完全还原**——导出稿与原稿的段落切分可能有细微差异。
8. **模型热切换不生效**：更换 embedding/分段模型后需重启后端；bge-m3 切换还会触发 kb_chunks 维度重建。
9. `api_key` 库内明文，keytar 是设计中的后续项（见《配置项参考》§4）。
10. 模型下载进度是**文件级粒度**（每个文件完成跳一次），不是字节级。
11. **electron-builder 在精简 PATH 的环境（如某些沙箱/CI）报 `spawn powershell.exe ENOENT`**：把 `C:\Windows\System32\WindowsPowerShell\v1.0` 加进 PATH 再跑 `npm run dist` 即可（npm 的 node 模块收集器内部要调 powershell）。

## 8. 代码风格约定

- 中文注释与 docstring；关键实现标注设计文档出处（如 `# §5.2④`）。
- 从旧版（`legacy/code/` 的 `main.py` / `src/` 树）移植的逻辑注明「移植旧版 xxx」。
- 每个里程碑的 `docs/dev/M*.md` 记录了当时的**设计取舍与踩坑**——改对应模块前先读它。
- 后端：FastAPI 路由层薄、逻辑在 `pipeline/` `rag/` `llm/`；数据库访问经 `core/db.py` 的 engine + sqlmodel Session。
- 前端：页面在 `pages/`、可复用 UI 在 `components/ui/`（手工 shadcn 风格，不引 radix）；状态只在 zustand store，组件不直接维护服务端数据。
