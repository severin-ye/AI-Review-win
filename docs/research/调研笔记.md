## 1. 句子分割模型：SaT `sat-3l-sm`（用户说的"MIT 小模型"）

**确认身份**：`segment-any-text/sat-3l-sm`，wtpsplit 库的 SaT（Segment Any Text）模型。
论文：Frohmann et al., "Segment Any Text: A Universal Approach for Robust, Efficient and Adaptable Sentence Segmentation", EMNLP 2024 (arXiv:2406.16678)。

**HF 官方页核实**（2026-07-17 访问）：
- License: **MIT** ✅
- 参数量 0.2B，F32；支持 **85 种语言**；有 ONNX、Safetensors 格式
- 月下载量 ~51.9 万；任务类型 Token Classification

**用法（CuraView 项目验证过的模式）**：
```python
from wtpsplit import SaT
sat = SaT("sat-3l-sm")          # 首次自动从 HF Hub 下载到 ~/.cache/huggingface
sentences = sat.split(text)     # list[str]
```
- CuraView 中封装为 LangChain `@tool split_sentences(text) -> List[Tuple[int, str]]`，**过滤 len(s) >= 10 的碎片**（对应 config `min_sentence_length: 10`）
- 模块级单例 `_load_model()` + `ensure_model_loaded()` 预加载模式（批量处理时模型只加载 1 次，10 患者省 ~90 秒）
- **回退**：模型加载失败时用正则 `r'([。!?;；\n]|(?<!\d)\.(?!\d))'` 规则分句
- CuraView 无下载脚本/无镜像配置，靠 wtpsplit 首次使用自动拉取；项目主要在 Linux 服务器跑

## 2. 混合检索（RAG + 关键词）

**业界标准方案：LangChain EnsembleRetriever = BM25Retriever + 向量检索器，RRF 融合**（多篇 2025-2026 实践文一致确认，掘金/SegmentFault/ arXiv:2510.08610）。

```python
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
bm25_retriever = BM25Retriever.from_documents(chunks); bm25_retriever.k = 3
ensemble = EnsembleRetriever(retrievers=[bm25_retriever, vector_retriever], weights=[0.5, 0.5])
```

- RRF 公式：`score(d) = Σ 1/(k + rank_i(d))`，**k 常取 60**
- 权重常见 0.5/0.5 或向量偏高（0.7/0.3）；专有名词多的语料调高 BM25
- 中文 BM25 需要分词器（jieba）传给 `preprocess_func`
- 本设计采用用户指定：**每句 向量 3 条 + 关键词 3 条**，去重后 ≤6 条证据
- 可选增强：交叉编码器重排序（bge-reranker，~1.3GB）——暂不做，留扩展

## 3. 查询重写（Multi-Query）

**业界数据点**：
- LangChain `MultiQueryRetriever`：默认让 LLM 生成 **3** 个改写
- RAG-Fusion（Rackauckas 2024）：通常 **4~5** 个 + RRF 融合
- GEO Community 实践指南（2026-01）：**3~6 条足够**，更多说明索引/切块有问题；每改写保守 k=5-10，合并去重，记录每条结果来自哪个改写
- DMQR-RAG（arXiv:2411.13154）：改写要多樣化；**改写数量不是越多越好**（引入噪声），自适应选择策略；保留原始查询入池
- 成本参考（ailog.fr）：Multi-query 延迟 ~3x、成本高，质量 +15~25%

**本设计决策**：用户指定 5~10 条 → **默认 8，可配置 5-10**；其中保留 1 条原句；一次 LLM 调用生成全部问题（JSON 数组）；每条问题并行打向量路+BM25 路；每路内部跨问题 RRF 融合去重；最终各路取 top-3。

## 4. Electron + Python sidecar 架构（2025 主流）

**验证案例：腾讯 TencentCloudADP/youtu-tip**（S 级来源）：
- Desktop: Electron + React + Tailwind + Vite
- Backend sidecar: **Python 3.11 + FastAPI + httpx，PyInstaller 打包**
- 目录：`electron/`（主+渲染+预加载）、`python/`（FastAPI app、services、Pydantic 模型、PyInstaller 配置）

**前端现代栈**（electron-vite 官方 + 2025 实践）：
- `npm create @quick-start/electron@latest`（electron-vite v3，需 Node 20.19+/22.12+，Vite 5+）
- 主进程/预加载 → CJS；渲染器 dev 转 ESM 支持 HMR，prod 打 IIFE
- Tailwind v4：`npm i -D tailwindcss @tailwindcss/vite`，vite 插件接入，CSS 顶部 `@import "tailwindcss";`
- shadcn/ui：需要 `@/*` alias（jsconfig/tsconfig paths）；其 CLI 只认根目录 vite.config.*（electron 项目需复制 electron.vite.config 为 vite.config.js 或手动配置）
- 主进程职责：窗口、原生对话框、IPC（ipcMain.handle）、应用生命周期
- 预加载：contextBridge.exposeInMainWorld 暴露受限 API

**本项目环境注意**（Windows 工作机）：
- `node` v24.15.0 可用，但 Git Bash 中 **`npm` 不在 PATH，必须用 `npm.cmd`**（v11.12.1）
- node 为 Kimi 桌面内置运行时；无 corepack/pnpm
- 根目录 `.npmrc` 配 `ELECTRON_MIRROR=https://npmmirror.com/mirrors/electron/` 防 Electron 二进制下载失败

## 5. 向量库与 Embedding 选型

- **LanceDB**：嵌入式、无服务、列存；CuraView GraphRAG 同款（`vector_store: lancedb`）；比 Chroma 轻 —— 选用
- **BGE-M3**（BAAI/bge-m3，MIT）：中英双语强，替代旧 text2vec-base-chinese；~2.2GB 下载 —— 默认选用，设置中可切换轻量 bge-base-zh-v1.5（~400MB）

## 6. 设计决策汇总（用户未否决的默认值）

| 问题 | 决策 |
|---|---|
| LLM 接入 | OpenAI 兼容协议一套（base_url 可配，通吃 OpenAI/DeepSeek/通义/vLLM）|
| Embedding | BGE-M3 |
| 旧 HMAC 每日密钥验证 | 废弃 |
| 旧代码 | 保留不动，新代码在 `electron/`、`renderer/`（不是 `src/`，与旧 Python 冲突）、`backend/` |
| 审校粒度 | block（多句，章节分组 ~1000 字），证据按句附着 |
| UI 语言 | 中文 |
| 结构化输出 | response_format=json_schema（fallback tool calling）；corrections 含 error_type/severity/evidence_ids/explanation |
| 持久化 | SQLite + SQLModel；数据目录 dev=`backend/.data/`，prod=`%APPDATA%/ai-review/` |
| 表格处理 | 沿用旧版占位符方案 `[{表格不予审校_N}]`，导出还原 |
| 导出 | 双版本：清洁版 + 留痕版 `｛～原文:x AI:y～｝` |
